Abstract

Zero-knowledge virtual machines (zkVMs) represent one of the most consequential innovations in applied cryptography since the invention of public-key encryption. By enabling any program written in a general-purpose language to produce a cryptographic proof of correct execution, zkVMs fundamentally reframe the trust model of distributed computation: rather than requiring every party to re-execute every computation, a single succinct proof — verifiable in milliseconds — suffices. This paper provides a comprehensive treatment of zkVM technology: its theoretical foundations in zero-knowledge proof theory, its architectural components (instruction set architecture, arithmetization, prover, verifier, recursion), the dominant proof systems deployed in production (STARKs with FRI, SNARKs with Groth16 and PLONK), and the full landscape of leading implementations including RISC Zero, Succinct SP1, a16z Jolt, StarkWare Cairo/Stwo, ZKsync Airbender, Nexus zkVM, and others. We analyze performance benchmarks, the zkVM trilemma of speed/security/generality, the distinction between zkVMs and zkCoprocessors, and the full spectrum of applications ranging from Ethereum Layer 2 rollups to cross-chain bridges, privacy-preserving identity, verifiable AI inference, and real-time Layer 1 block proving.

The paper concludes with an original practical implementation section in which the authors build three working zkVM programs directly on the SP1 v6.1.0 source codebase: a ZK Password Verifier (SHA-256 hash preimage proof), a ZK Age Verifier (threshold proof without value revelation), and a ZK Human Verifier (multi-criteria selective disclosure with five test subjects). These implementations are used to derive concrete observations about the host/guest programming model, precompile performance advantages, and the structural enforcement of the zero-knowledge property in production zkVM systems.

Keywords: zero-knowledge proofs, zkVM, STARK, SNARK, RISC-V, arithmetization, verifiable computation, zkRollup, SP1, RISC Zero, Jolt, recursive proofs, Ethereum scaling, selective disclosure, practical implementation.

§ 1

Introduction and Problem Statement

1.1 The Fundamental Problem of Verifiable Computation

Consider a fundamental problem in distributed systems: a client outsources a complex computation to an untrusted server. How can the client be certain the server returned the correct result, without re-executing the computation? The naïve answer — re-execution — defeats the purpose of outsourcing entirely. For decades, this problem had no satisfying cryptographic solution. Clients either trusted the server, employed redundant execution (expensive), or used probabilistic checks (incomplete).

This problem is not merely academic. It sits at the heart of blockchain scalability. Every Ethereum full node today re-executes every transaction ever submitted to the network. As of 2026, this means billions of nodes collectively performing trillions of redundant operations — an extraordinary waste that caps the network's throughput. Ethereum processes roughly 15 transactions per second; Visa processes roughly 24,000. The gap is not fundamentally architectural — it is a verification problem.

The same problem appears in privacy-preserving applications. A user wants to prove they satisfy some criterion (age, creditworthiness, citizenship, KYC status) without revealing the underlying data. How can they prove a fact without revealing the secret that makes that fact true?

And in artificial intelligence: as models become commercial infrastructure, a critical question arises — how does a user know that a model provider actually ran their claimed model on their claimed inputs, without bias, tampering, or substitution? The answer, in all these cases, is the same: a succinct cryptographic proof of correct computation.

1.2 The Zero-Knowledge Virtual Machine as a Solution

A zero-knowledge virtual machine (zkVM) is a machine that executes arbitrary programs and produces a compact cryptographic receipt — a zero-knowledge proof — attesting to the correctness of that execution. The verifier can check this proof in milliseconds regardless of how long the original computation took. This seemingly simple idea has profound implications:

  • It decouples computation from verification. Computation can be expensive; verification need not be.
  • It enables trustless delegation. A server can prove it did the work correctly without the client trusting the server.
  • It enables privacy. Proofs can be generated without revealing the inputs that produced them.
  • It scales blockchains. A single proof can attest to millions of transactions, verified on-chain for a fraction of the cost of re-execution.
  • It democratizes ZK. Developers need not understand advanced cryptography — they write normal code, and the zkVM handles the proof generation.
Key Insight

The "zk" in zkVM is a misnomer in most production deployments. The essential property is succinctness — proofs are small and fast to verify, regardless of computation size. Zero-knowledge (hiding inputs) is optional and often not enabled. The "zk" prefix has become standard industry terminology, adopted by systems where privacy is secondary or absent. This paper follows that convention.

1.3 Historical Context

The concept of a zkVM traces to the 2013 paper "SNARKs for C: Verifying Program Executions Succinctly and in Zero Knowledge" by Ben-Sasson, Chiesa, Genkin, Tromer, and Virza, which introduced TinyRAM — a minimalist RISC machine with 29 opcodes designed to be proved efficiently. TinyRAM established the foundational research direction and spawned a decade of innovation. Today's production systems are orders of magnitude more performant, but the underlying insight remains the same: model computation as execution on a simple machine, then prove that execution.

Between 2018 and 2022, practical zkVMs began appearing in Ethereum's Layer 2 ecosystem — first as zkEVMs optimized for Ethereum compatibility, then as general-purpose RISC-V-based systems capable of proving arbitrary computations. By 2025, the field had matured to the point where zkVMs were proving Ethereum blocks in under a minute; by late 2025, under 12 seconds — the Ethereum Foundation's threshold for "real-time proving." The trajectory points toward zkVMs becoming as invisible as TLS: always running, never noticed, but foundational to the trust architecture of the web.

§ 2

Background: Zero-Knowledge Proofs

2.1 Definitions and Formal Properties

A proof system for a language L consists of an interaction between a prover P and a verifier V. The prover attempts to convince the verifier that some statement x belongs to L. A zero-knowledge proof system satisfies three properties:

Completeness. If x ∈ L and P is honest, then V accepts with overwhelming probability. An honest prover can always convince an honest verifier.

Soundness. If x ∉ L, then no computationally bounded cheating prover P* can convince V to accept, except with negligible probability. A dishonest prover cannot forge proofs.

Zero-Knowledge. If x ∈ L, then V learns nothing from the proof beyond the truth of the statement. Formally, there exists a polynomial-time simulator S such that the output of S is computationally indistinguishable from a real proof transcript.

For verifiable computation, a fourth property is critical:

Succinctness. The proof size and verification time are sublinear (ideally polylogarithmic or constant) in the computation size. Without succinctness, verification is as expensive as re-execution, defeating the purpose.

Systems satisfying completeness, soundness, and succinctness — without necessarily satisfying zero-knowledge — are called Succinct Arguments of Knowledge (SAKs) or informally SNARKs/STARKs. Most production zkVMs prioritize succinctness over strict zero-knowledge.

2.2 From Circuits to General Computation

Early ZK proof systems operated over arithmetic circuits — directed acyclic graphs where wires carry field elements and gates perform addition or multiplication over a finite field F. Any computation can be expressed as an arithmetic circuit, but circuits are inherently static: the circuit structure must be fixed before a proof is generated. This creates a critical limitation: you cannot prove a program that branches on secret data, loops a variable number of times, or accesses memory in an unpredictable order, without an exponential blowup in circuit size.

The solution is to model computation differently. Rather than proving a circuit, a zkVM proves the execution trace of a program running on an abstract machine. This trace is a table: each row records the machine state at one clock cycle (program counter, registers, memory access, opcode). Proving the execution trace is equivalent to proving the program ran correctly, and the table structure naturally handles branches, loops, and dynamic memory.

2.3 SNARKs and STARKs: The Two Families

The two dominant families of proof systems in production zkVMs have different tradeoff profiles:

SNARKs  — Succinct Non-Interactive Arguments of Knowledge
  • Extremely small proofs (~200 bytes for Groth16)
  • Fast on-chain verification (<1ms)
  • Mature tooling and ecosystem
  • Require a trusted setup (KZG ceremony)
  • Not post-quantum secure
  • Rely on elliptic curve pairings (BLS12-381, BN254)

Examples: Groth16, PLONK, Halo2 (no trusted setup via accumulation), Nova

STARKs  — Scalable Transparent Arguments of Knowledge
  • Transparent — no trusted setup required
  • Post-quantum secure (hash-function based)
  • Naturally fit step-by-step execution traces (AIR)
  • Highly parallelizable proof generation
  • Larger proofs (~100–500 KB)
  • Slower on-chain verification if not wrapped

Examples: FRI-STARKs (Starkware), Plonky3, Circle STARKs (Stwo), Boojum

Common Pattern in Production

Many production zkVMs combine both: a STARK prover generates a proof quickly (taking advantage of transparency and parallelism), and then a final SNARK wrapper (typically Groth16) compresses that proof to ~200 bytes for cheap on-chain verification. This gives the transparency of STARKs with the succinctness of SNARKs — at the cost of an additional wrapping step.

What is a zkVM?

A zero-knowledge virtual machine is a system that: (1) executes a program expressed in some instruction set architecture, (2) records the full execution trace, and (3) generates a cryptographic proof that the execution occurred correctly with respect to the specified inputs and program. A verifier with access only to the proof, the program, and the public inputs can efficiently confirm the result's correctness without re-executing the program.

The key distinction from a traditional virtual machine is the addition of the cryptographic proof layer. A traditional VM executes code and returns results; the results are trustworthy only because you trust the machine. A zkVM executes code and returns both results and a proof; the results are trustworthy because the proof is verified mathematically, regardless of whether you trust the machine.

This distinction is architecturally significant. The zkVM must model computation in a form that is amenable to algebraic proof techniques. This shapes every design decision: the choice of instruction set, the representation of memory, the handling of control flow, and the selection of the underlying proof system.

3.1 The Host/Guest Model

Most production zkVMs separate the application into two components:

The guest program is the code that gets proved. It runs inside the zkVM, and its execution trace is used to generate the proof. The guest has access to special I/O mechanisms to receive secret inputs (private witness) and emit public outputs (journal).

The host program is the outer environment that controls the zkVM. The host loads the guest program, provides inputs, receives outputs, and manages the proving process. The host is not proved — it is trusted code running on the developer's infrastructure.

This separation allows developers to write complex orchestration logic in the host without proving it, while isolating the critical computation that needs trustless verification into the guest. The guest program is compiled to an ELF binary (in RISC-V-based systems), which the executor runs and traces before the prover generates the receipt.

A zkEVM is a specialized zkVM whose instruction set is the Ethereum Virtual Machine — it proves Ethereum smart contract execution. zkEVMs have been classified into types based on their compatibility with the EVM:

§ 4

Architecture: Deep Dive

4.1 Instruction Set Architecture (ISA)

The ISA is the most consequential design decision in building a zkVM. It defines the universe of programs the system can execute and prove, and fundamentally determines the complexity of the proof system.

RISC-V has emerged as the dominant ISA for general-purpose zkVMs. Developed at UC Berkeley in 2010, RISC-V is an open-standard, minimal instruction set with just 47 base integer instructions (RV32I). Its properties that make it ideal for zkVM use are: open standard (no licensing), LLVM support (compile C, C++, Rust, Go directly), minimal instruction count (fewer opcodes to arithmetize), and a large ecosystem of existing tooling. RISC Zero, SP1, Nexus, ZKM, and Airbender all target RISC-V.

ZK-native ISAs are instruction sets designed from the ground up for proof-efficiency. Cairo (StarkWare) and Miden (Polygon) are the leading examples. Cairo's VM uses a special memory model — write-once, non-deterministic reads — that is extremely efficient to arithmetize. ZK-native ISAs can achieve dramatically lower constraint counts than RISC-V for equivalent computations, but require developers to use a new language or compiler toolchain, fragmenting ecosystem compatibility.

EVM targets Ethereum compatibility. The EVM is notoriously difficult to arithmetize: 256-bit word size (requiring range checks in 32-bit proof fields), variable-length stack operations, expensive storage access patterns, and complex opcode semantics. Every zkEVM team has developed custom arithmetization strategies to manage this complexity.

4.2 Execution and Trace Generation

The executor is a standard virtual machine that interprets the compiled bytecode. For RISC-V systems, this means running an ELF binary through the standard fetch-decode-execute cycle. The critical addition relative to a normal VM is trace recording: every state transition is captured in a structured table.

The execution trace is a matrix where each row corresponds to one clock cycle. Columns record:

Row_i = (pc_i, opcode_i, rs1_i, rs2_i, rd_i, imm_i, mem_addr_i, mem_val_i, clk_i, ...)

where pc is the program counter, opcode is the instruction, rs1 and rs2 are source registers, rd is the destination register, imm is an immediate value, and clk is the cycle counter. The trace also includes auxiliary columns for memory access, range checks, and chiplet I/O.

The trace is then partitioned into tables handled by different components of the system — the CPU table, the memory table, the arithmetic table, and so on. These tables are later proved using polynomial constraints (see §4.3) and connected via lookup arguments (see §5.4).

4.3 Arithmetization

Arithmetization is the process of converting the execution trace into polynomial constraints that can be verified by a proof system. This is arguably the most technically subtle component of a zkVM, and the source of most performance differences between systems.

R1CS (Rank-1 Constraint System) is the arithmetization scheme used by older SNARKs (Groth16). Every constraint takes the form:

(a · x) ⊙ (b · x) = (c · x)

where x is the witness vector and a, b, c are selector vectors. R1CS is expressive but requires converting control flow into multiplication chains, which is expensive. It is rarely used in modern zkVMs.

PLONKish (PLONKish arithmetization) used by Halo2 and related systems. Generalizes R1CS to support custom gates — rather than only multiplication gates, the developer can define arbitrary polynomial constraints per row. This is more expressive and allows more efficient representation of operations like range checks, bitwise operations, and lookups.

AIR (Algebraic Intermediate Representation) is the dominant arithmetization for STARK-based zkVMs. An AIR defines a set of polynomial constraints that relate adjacent rows of the execution trace:

P(row_i, row_{i+1}) = 0 for all i in [0, N-1]

This "next-row" constraint naturally models state transitions in a sequential machine: the constraint can check that the program counter incremented correctly, that register writes were valid, that memory accesses were consistent. AIR is particularly well-suited to zkVMs because the step-by-step structure of program execution maps directly onto the adjacent-row structure of AIR constraints.

4.4 Memory Checking: The Hard Problem

Memory is one of the most challenging aspects of zkVM design. A program may read and write to millions of memory addresses in unpredictable order. How do we prove that every read returns the most recently written value at that address?

The naïve approach would be to include the full memory state in every row of the execution trace. For a machine with 4 GB of addressable memory and millions of execution steps, this is astronomically expensive.

Offline memory checking (introduced by Blum et al., 1991, and adapted for zkVMs) is far more efficient. The key insight is that we only need to prove consistency between accesses, not between states. The zkVM maintains two logs:

  • A write log W: records every write (addr, val, clk) plus initial memory values.
  • A read log R: records every read (addr, val, clk).

Correctness requires: every read at (addr, clk) must see the value written in the most recent write to addr at some clk' < clk. This is equivalent to saying that R and W, when sorted by (addr, clk), form a consistent timeline. This multiset equality can be checked using a multiset hash or a grand product argument — polynomial operations that are cheap to prove.

Memory Timestamp Trick

Modern zkVMs (SP1, Nexus, Airbender) use an explicit timestamp column in the trace. Each read records the timestamp of the previous write to that address (clk_prev) and the current timestamp (clk_cur). A constraint enforces clk_prev < clk_cur, and a separate multiset argument proves that every (addr, val, clk_prev) tuple in the read log appears in the write log. This decomposes the global memory consistency problem into local row-level constraints plus one multiset check.

4.5 Chiplets and Precompiles

Some operations are prohibitively expensive to prove directly in the main CPU trace. Cryptographic hash functions (SHA-256, Keccak-256), elliptic curve point multiplication, RSA exponentiation, and large-integer arithmetic all require tens of thousands of constraints per operation in a naïve AIR formulation. zkVMs address this with two techniques:

Chiplets (used by Miden, originally inspired by Polygon's arithmetization work) are specialized sub-circuits that handle expensive operations independently. The main CPU trace can invoke a chiplet by writing the inputs to a shared bus and reading back the outputs. The chiplet produces its own trace, proved separately, with the correctness of the I/O enforced by a lookup argument between the CPU trace and the chiplet trace.

Precompiles (used by SP1 and RISC Zero) are hand-written, optimized circuits for specific operations that replace the naive RISC-V implementation of those operations. When the guest program calls, for example, sha256(data), the zkVM detects this call and routes it to the SHA-256 precompile — a hand-crafted AIR circuit that computes SHA-256 in far fewer constraints than the RISC-V instruction-by-instruction implementation would require. SP1's precompiles reduce cycle counts by 5–10× for workloads dominated by hashing and elliptic curve operations.

4.6 Recursion and Proof Composition

A critical challenge in zkVM design is handling programs of unbounded length. A STARK proof for N execution steps has size O(log² N) — polylogarithmic, but still growing. For a program running for billions of cycles, even a polylogarithmic proof becomes impractically large.

Recursive proof composition solves this. The key insight: the verifier for a proof is itself a computation, and that computation can be proved by another zkVM instance. This creates a "proof of a proof" — a single fixed-size proof that attests to the correctness of the entire chain of computation, regardless of how long the original program ran.

In practice, zkVMs use continuations: the program is divided into fixed-size segments (typically 16M or 64M cycles). Each segment produces a STARK proof. These proofs are then recursively aggregated — each aggregation step proves that the previous proofs were correct — producing a single final proof. RISC Zero and SP1 both implement continuation-based recursion, enabling proof of programs of arbitrary length.

Proof_final = Prove(Verify(Proof_1) ∧ Verify(Proof_2) ∧ ... ∧ Verify(Proof_N))

An alternative approach is folding schemes (Nova, SuperNova, HyperNova), which "fold" multiple instances of a relation into a single instance incrementally, without requiring a full recursion step. Nexus and Novanet use folding schemes, which have excellent theoretical properties for very long computations but currently underperform STARK-based systems in practice due to large proof sizes.

§ 5

Proof Systems in Depth

5.1 Polynomial Commitment Schemes

Every modern proof system is built on a polynomial commitment scheme (PCS) — a protocol that allows a prover to commit to a polynomial and later prove that the polynomial evaluates to a specific value at a specific point, without revealing the polynomial itself. The PCS is the cryptographic engine underneath the proof system, and the choice of PCS determines many fundamental properties.

KZG commitments (Kate-Zaverucha-Goldberg, 2010) use elliptic curve pairings to produce constant-size commitments and proofs. They underlie PLONK, Halo2, and many SNARKs. They require a trusted setup (structured reference string from a powers-of-tau ceremony) and are not post-quantum secure.

FRI (Fast Reed-Solomon IOPP) is a hash-function-based PCS that achieves near-linear proving time and polylogarithmic proof size with no trusted setup. FRI underlies all STARK systems. It works by repeatedly folding the polynomial into lower-degree polynomials and providing proximity proofs at each level via Merkle trees.

Brakedown and Binius are newer hash-based PCS schemes designed specifically for binary fields (F_2), enabling highly parallelizable proof generation on commodity hardware. Binius in particular has attracted attention for its potential to replace BabyBear-based systems with even faster provers.

5.2 FRI and STARKs: The Dominant Approach

The FRI-STARK pipeline is the dominant approach in production zkVMs (SP1, RISC Zero, Cairo/Stwo, Airbender). The high-level structure is:

  • Step 1 — Trace commitment: The execution trace is interpolated as a polynomial and committed via a Merkle tree over a low-degree extension field.
  • Step 2 — Constraint composition: All AIR constraints are combined into a single quotient polynomial Q(x) that equals zero on all rows of the trace if and only if all constraints are satisfied.
  • Step 3 — FRI proximity proof: The prover convinces the verifier that Q(x) is of sufficiently low degree (i.e., that it is actually a valid polynomial, not arbitrary data), using the FRI protocol. This is the computationally intensive step.
  • Step 4 — Query phase: The verifier sends random challenges; the prover opens the committed polynomials at those points and provides Merkle proofs of consistency.

The soundness of the STARK is determined by the number of queries and the rate of the Reed-Solomon code. More queries means higher soundness but larger proofs. Production systems balance this to achieve 128-bit security with reasonably compact proofs.

5.3 Plonky3 and the BabyBear Field

Plonky3, developed by Polygon and used as the proving backend for SP1, is a modular STARK proving framework designed for maximum performance. Its key innovation is the choice of field: the BabyBear prime, defined as p = 2³¹ − 2²⁷ + 1 ≈ 2.01 × 10⁹.

The BabyBear prime fits in a 32-bit machine word, meaning all field arithmetic can be performed without 64-bit overflow handling. This is a critical performance advantage: modern CPUs and GPUs have highly optimized 32-bit arithmetic pipelines, and Plonky3's field arithmetic runs at near-native speed. Older systems using the Goldilocks prime (64-bit) or BN254 (256-bit) are substantially slower per field operation.

BabyBear: p = 2^31 - 2^27 + 1 = 2,013,265,921 (fits in u32) Goldilocks: p = 2^64 - 2^32 + 1 (fits in u64) BN254: p ≈ 2^254 (requires multi-word arithmetic)

5.4 Lookup Arguments: The Jolt Innovation

Traditional STARK systems prove each RISC-V instruction by encoding its semantics as polynomial constraints. Even with chiplets, this requires tens of constraints per instruction. A fundamentally different approach is to use lookup arguments.

A lookup argument proves that every value in one table appears in another (pre-computed) table, without directly proving the computation. The Lasso lookup argument (used by Jolt) takes this to an extreme: for RISC-V instructions, the output of every instruction is pre-computed in a massive lookup table. Proving an instruction execution becomes proving that (input, output) appears in the table — a much cheaper operation than re-proving the computation directly.

The challenge is the size of the lookup table: for a 32-bit instruction set, the full table has 2⁶⁴ entries — obviously infeasible to store. Lasso uses polynomial techniques (multilinear extensions and sparse table representations) to compress these tables dramatically. The MLE (multilinear extension) of a sparse table can be computed and committed without materializing the full 2⁶⁴-entry table.

LogUp is a lookup argument used by SP1 and Plonky3. It represents lookup constraints as logarithmic derivatives and uses a grand product check. LogUp is highly efficient and well-suited to the chiplet/precompile architecture of SP1.

Plookup (used by PLONK-based systems) sorts the table and the witness together and checks that the sorted combined sequence is a valid interleaving. Less efficient than LogUp for large tables but simpler to implement.

Major Implementations

The zkVM landscape has evolved rapidly from academic prototypes to production-grade systems. The following are the leading implementations as of April 2026, evaluated across architecture, performance, maturity, and ecosystem.

SP1 (Succinct Processor One)
Succinct Labs · 2024–present
Production

SP1 is the developer-experience leader of the zkVM space and arguably the most widely deployed general-purpose zkVM as of 2026. Its design philosophy centers on making zero-knowledge proofs accessible to any Rust developer without requiring cryptographic expertise. Programs are compiled to RISC-V using the standard Rust toolchain; the developer never touches constraint systems or polynomial arithmetic.

SP1's proving backend is Plonky3, using AIR arithmetization over the BabyBear field with FRI polynomial commitments and LogUp lookup arguments. The system uses a precompile architecture where common cryptographic operations (SHA-256, Keccak, secp256k1, ed25519, BN254, BLS12-381) are replaced with hand-optimized circuits, reducing cycle counts by 5–10×. GPU acceleration via CUDA and CPU acceleration via AVX512/NEON are both supported. A prover network (Boundless) allows on-demand proof generation without local hardware.

7.1 Benchmarking Challenges

Comparing zkVM performance is notoriously difficult. Results depend on: hardware (CPU vs GPU, GPU model, RAM), whether hardware acceleration is enabled (CUDA, AVX), proof mode (STARK only vs STARK + SNARK wrapper), hash function selection (Poseidon vs SHA-256 vs Keccak), continuation depth (single proof vs recursive aggregation), and the specific workload (Fibonacci vs SHA-256 vs ECDSA vs Ethereum block). 7.5 Compiler Optimization and zkVM Performance

A 2025 study systematically evaluated the impact of LLVM optimization passes on SP1 and RISC Zero across 58 benchmarks. The findings are informative: standard LLVM optimization levels improve zkVM performance by over 40% compared to unoptimized code, but far less than on native CPUs — because LLVM optimizations target hardware features (instruction pipelining, cache behavior, branch prediction) that do not exist in the proof model. The most impactful pass was inline (22.4% gain on both systems), because function call overhead in a zkVM is higher than on native hardware. The most harmful pass was licm (loop-invariant code motion), which increases dynamic instruction count by moving code out of loops into preambles that the zkVM cannot optimize away.

§ 8

Applications and Use Cases

8.1 Blockchain Scaling: zkRollups

The original and still most deployed use case for zkVMs is Layer 2 scaling. A zkRollup executes transactions off-chain, produces a validity proof that the transactions were processed correctly, and posts only this proof (plus compressed transaction data) to Ethereum Layer 1. The L1 verifier contract checks the proof and updates state accordingly — without re-executing any transaction.

This achieves dramatically higher throughput and lower transaction costs. zkSync processes thousands of transactions per second (vs. Ethereum L1's ~15 TPS) at a fraction of the cost. The validity proof provides cryptographic finality — unlike optimistic rollups, which require a 7-day challenge window, zkRollup finality is achieved as soon as the proof is verified on L1 (typically within minutes).

8.2 Layer 1 Real-Time Proving

The most ambitious application of zkVMs is proving Ethereum L1 execution in real time — generating a proof for each block as it is produced, so that validators can verify blocks by checking a proof rather than re-executing them. This would allow Ethereum's validator set to include extremely lightweight verifiers, dramatically improving decentralization and enabling a 10–100× increase in L1 throughput.

The Ethereum Foundation defines the target as under 12 seconds per block. By October 2025, Brevis Pico Prism became the first zkVM to consistently achieve this on a 45M gas block (the maximum block size). The trajectory from 35 minutes (2024) to 6.9 seconds (2025) represents a 300× improvement in 12 months — one of the most rapid performance scaling curves in applied cryptography.

8.3 Cross-Chain Bridges and Light Clients

Trustless bridges between blockchains require proving the state of one chain to another. Traditional bridges use multi-sig federations (trusted, hackable) or optimistic mechanisms (slow). zkVM-based light clients can prove the validity of another chain's block headers in a succinct proof, enabling trustless cross-chain state verification.

Celestia's DA attestation proofs, IBC light client proofs, and EigenLayer's AVS (Actively Validated Services) all use this pattern. RISC Zero's zkVM and SP1 are the primary proving backends for these applications, with Polygon and Avail using SP1 for their cross-chain infrastructure.

8.4 Privacy-Preserving Identity and Compliance

zkVMs enable a powerful pattern for identity and compliance: prove a fact about yourself (age, residency, creditworthiness, KYC status) without revealing the underlying data. The user generates a proof using their private data as the witness; the verifier sees only the proof and can confirm the statement is true without learning any sensitive information.

Applications include: zkKYC (prove you passed identity verification without revealing your passport), privacy voting (prove your vote is valid without revealing how you voted), anonymous credentials (prove organizational membership without revealing identity), and private DeFi (prove you are creditworthy for uncollateralized lending without revealing your balance sheet).

8.5 Verifiable AI Inference (ZKML)

As AI systems become critical infrastructure, a new trust problem emerges: how does a user know that a model provider actually ran the claimed model with the claimed weights on the claimed input, without tampering? Zero-knowledge machine learning (ZKML) uses zkVMs to prove the correct execution of ML inference.

Practical ZKML faces significant challenges: neural network inference involves billions of multiply-accumulate operations, making proving extremely expensive. However, progress has been rapid. RISC Zero's zkVM has been used to prove transformer inference; Giza (built on Cairo) specializes in ZKML for on-chain AI. The primary application today is proving the output of smaller models (classification networks, recommendation engines) rather than LLMs, but the trend is toward proving increasingly capable models as prover performance improves.

8.6 On-Chain Gaming and Covert Computation

zkVMs enable on-chain games with hidden state — chess, poker, strategy games — where moves can be proved valid without revealing the player's information. This is a distinct use case from privacy coins: the goal is not financial privacy but game-theoretic integrity. A player can prove "I moved my knight legally" without revealing where all their pieces are. Projects like Dark Forest (Ethereum, early ZK gaming) pioneered this category.

8.7 Supply Chain and Data Provenance

zkVMs can prove facts about data pipelines without revealing the underlying data. A pharmaceutical company can prove that a drug batch was stored at the correct temperature throughout its supply chain, without revealing commercially sensitive routing or supplier information. A financial institution can prove that its trading algorithms executed within regulatory bounds, without revealing proprietary strategy. The "proof of correct computation over private data" pattern is general and applies across industries.

§ 9

The zkVM Trilemma and Open Challenges

9.1 The zkVM Trilemma

Analogously to the blockchain scalability trilemma (decentralization, security, scalability), zkVM designers face a trilemma among three desirable properties:

⚡
Speed
Fast proof generation and verification. Sub-second proving for short programs; real-time for block-length computations. GPU parallelism, efficient fields, hardware acceleration.
🔐
Security
Sound proofs with high security parameters. No trusted setup. Post-quantum resistance. Formally verified circuits. Resistance to implementation bugs and soundness errors.
🌍
Generality
Arbitrary programs in any language. Existing libraries. No custom DSL. Full ISA support including complex memory access patterns, floating point, system calls.

ZK-native ISAs (Cairo, Miden) sacrifice generality for speed — they achieve excellent proving performance but require developers to write in unfamiliar languages and cannot use existing software libraries. General-purpose RISC-V systems (SP1, RISC Zero) sacrifice some speed for maximum generality — any Rust program can be proved, but at higher prover cost than a purpose-built circuit. Lookup-based systems (Jolt) sacrifice production readiness and GPU efficiency in pursuit of theoretical elegance that may eventually unlock future speed gains.

No current system achieves all three simultaneously. The trilemma is a useful lens for understanding design decisions: every architectural choice is a tradeoff within this space.

9.2 Open Challenges

The memory bottleneck. As programs become more complex and guest memory grows (RISC Zero supports up to 3 GB), the cost of memory checking operations grows proportionally. Efficient memory checking for very large programs (hundreds of gigabytes) remains an open research problem.

Floating-point operations. Standard RISC-V includes a floating-point extension (RV32F/D) that most zkVMs do not support. Proving IEEE 754 floating-point semantics is extremely expensive due to the complex edge cases (NaN propagation, denormalized numbers, rounding modes). This limits zkVMs for scientific computing applications.

Soundness bugs and circuit verification. The complexity of zkVM circuits creates opportunities for subtle soundness errors — bugs where the prover can generate a valid-looking proof for an invalid computation. The history of zkEVMs has several examples of such bugs discovered post-deployment. Formal verification of circuits is the gold standard solution but remains incomplete for most systems.

Front-end field mismatch. RISC-V programs operate on 32-bit or 64-bit integer words, but zkVM proof arithmetic operates over prime fields (BabyBear: ~2×10⁹). When a RISC-V instruction computes a result that would overflow a 32-bit integer (wrapping modular arithmetic), the zkVM must simulate this behavior in a field where the natural modulus is different. This field mismatch requires careful constraint engineering and is a source of both performance overhead and potential bugs.

Developer tooling and debugging. Debugging zkVM programs is fundamentally harder than debugging normal programs. Standard debuggers cannot step through the proof circuit. Errors in ZK programs can be non-deterministic, constraint-system-level, or only apparent at proof generation time. The ecosystem of zkVM-aware debuggers, profilers, and testing frameworks is nascent.

Prover decentralization. Most zkVM applications today rely on centralized or semi-centralized provers. Decentralized prover networks (Boundless, Aligned) are being built, but ensuring that the prover market is competitive, resistant to collusion, and achieves timely proof generation at scale remains an engineering and incentive design challenge.

A zkCoprocessor is a pre-built ZK proving service for a specific class of computations. Rather than requiring the application developer to write and prove arbitrary guest code, the zkCoprocessor exposes a typed API: "prove that account X held at least Y ETH at block Z" or "prove that this price feed was above threshold T for N consecutive blocks." Leading zkCoprocessors in the Ethereum ecosystem as of 2026:

  • Brevis: Dual architecture — general zkVM (Pico) for complex tasks plus zkCoprocessor for DeFi/RWA/risk management. First to achieve EF RTP standard.

  • Lagrange: Cross-chain computation with a ZK + Optimistic hybrid architecture, optimized for cross-chain data availability.

10.2 Which to Choose

The choice between zkVM and zkCoprocessor is primarily driven by the novelty and specificity of the computation to be proved. For computations that fit a well-defined template (historical state queries, liquidity analytics, risk scoring, identity verification), a zkCoprocessor offers faster integration, lower cost, and better performance — the specialized circuit is far more efficient than a general-purpose RISC-V execution environment. For novel computations (custom algorithms, AI inference, new cryptographic protocols, supply chain logic), only a zkVM can provide the required generality.

The two paradigms are not mutually exclusive and are increasingly converging. Systems like Brevis integrate both: a general-purpose zkVM backend for complex arbitrary computation, plus a specialized coprocessor interface for high-frequency, production DeFi use cases.

§ 11

Security Considerations

11.1 Soundness and its Failure Modes

The security of a zkVM rests on the soundness of its proof system: no cheating prover can generate a valid proof for an invalid execution. Soundness failures — commonly called soundness bugs — allow a malicious prover to convince a verifier of a false statement, potentially stealing funds in a zkRollup or forging identity proofs.

Soundness bugs in zkVM circuits typically arise from:

  • Under-constrained polynomials: A constraint is missing, allowing the prover to set a wire to an incorrect value that still satisfies all stated constraints.
  • Range check gaps: In systems where computations are over a large field but model 32-bit arithmetic, missing range checks allow field elements larger than 2³² to be used where 32-bit integers are expected.
  • Memory consistency errors: Bugs in the multiset argument for memory checking that allow reads to return incorrect values.
  • Cross-table lookup mismatches: Inconsistencies between the main CPU trace and chiplet/precompile traces that are not fully enforced by the lookup argument.

11.2 Formal Verification

The complexity of zkVM circuits makes manual auditing insufficient. Formal verification — mathematically proving that a circuit implements its specification — is the gold standard. RISC Zero uses the ZIRGEN language (an internal DSL for circuit specification) and has completed deterministic verification for most RISC-V base circuits. Nexus has invested in formal specification of its entire architecture. The Jolt paper includes formal proofs of the Lasso lookup argument's soundness.

Tools for zkVM circuit verification include: the Lean4 proof assistant (used for Jolt's Lasso semantics verification, published at FC '25), Coq-based approaches for arithmetic circuits, and domain-specific SMT-solver based tools like the 2025 Arguzz fuzzer for zkVM soundness testing.

11.3 Trusted Setup and Transparency

SNARK systems using KZG polynomial commitments require a trusted setup ceremony — a multi-party computation that generates a structured reference string (SRS). If all participants of the ceremony collude or are compromised, the SRS can be used to generate fake proofs. This is the "toxic waste" problem.

STARK systems (using FRI/Merkle-tree commitments) are transparent — no trusted setup is required. The verifier needs only a hash function specification. This is a significant security advantage for long-lived systems where the provenance of a setup ceremony may be hard to audit years later. All major zkVMs (SP1, RISC Zero, Cairo/Stwo, Airbender) use STARK-based backends for this reason, using Groth16 only as a wrapping step where the ceremony risk is manageable and accepted.

11.4 Post-Quantum Security

STARK systems are post-quantum secure: their security rests on the collision resistance of hash functions (SHA-256, Poseidon), which are not broken by quantum computers with known algorithms. SNARK systems using elliptic curve pairings (KZG, Groth16) are vulnerable to Shor's algorithm on a sufficiently powerful quantum computer. As quantum computing advances, the transition to post-quantum proof systems is a long-term security consideration. The STARK-based architecture of most production zkVMs provides this property by default, making their security guarantees durable against future quantum adversaries.

§ 12

Future Directions and Conclusion

12.1 Near-Term Developments (2026–2028)

Ethereum L1 integration. The Ethereum Foundation's roadmap ("The Verge") includes making every Ethereum validator a zkVM verifier — replacing re-execution with proof checking at the L1 level. This requires real-time proving (achieved in 2025 at the research level) plus integration with Ethereum's consensus and execution clients. The goal is to allow Ethereum to scale throughput dramatically without sacrificing decentralization.

Proof marketplace commoditization. RISC Zero's Boundless and similar platforms are building open markets for ZK proof generation. Any application can request a proof; any prover with sufficient hardware can compete to generate it. This commoditizes proof generation, drives down costs, and improves censorship resistance. The economics of proof markets — prover incentives, latency guarantees, MEV in the proof layer — are active research areas.

Hardware acceleration beyond GPU. FPGAs and ASICs designed specifically for STARK proving are in development. StarkWare and others have published FPGA implementations of FRI. Purpose-built ZK ASICs could achieve 10–100× speedups over GPU implementations, potentially enabling real-time proving on consumer hardware — a radical democratization of proof generation.

Binius and binary field proving. Binius, a polynomial commitment scheme over binary fields (F₂), promises dramatically faster field arithmetic by leveraging hardware-native bitwise operations. Several teams are prototyping Binius-based zkVMs; if the performance gains materialize, Binius may replace BabyBear-field STARKs as the dominant approach.

12.2 Long-Term Vision: zkVMs as Invisible Infrastructure

The trajectory of zkVM technology points toward a world where proof generation is as invisible and ubiquitous as TLS encryption. Today, every secure web request is encrypted — users never think about it. In a zkVM-enabled future, every computation delegated to an untrusted party would be accompanied by a proof of correctness — applications would never think about it either. The zkVM would be a layer in the infrastructure stack, invisible to application developers but foundational to the trust architecture of distributed systems.

This vision — ZK everywhere — is not science fiction. The rate of performance improvement in zkVM technology from 2022 to 2026 rivals Moore's Law: proving time for an Ethereum block has improved by roughly 3,000× (from ~100 hours equivalent to ~35 seconds) in four years. At this rate, within a decade, proving a block-length computation in under one second on consumer hardware is plausible.

12.3 Conclusion

Zero-knowledge virtual machines represent a convergence of three decades of theoretical work in interactive proof systems, polynomial commitments, and algebraic complexity theory with practical engineering driven by the demand for scalable, trustless computation in blockchain ecosystems. The result is a technology that is simultaneously: theoretically deep (resting on hardness assumptions with rigorous proofs), practically valuable (enabling Ethereum to process thousands of transactions per second with cryptographic finality), and socially significant (enabling privacy-preserving identity, compliance, and computation at scale).

The field is moving extraordinarily fast. The primary challenges — proving speed, developer experience, formal verification, and hardware accessibility — are all being actively addressed by well-funded teams and world-class cryptographers. The architectural patterns described in this paper — RISC-V ISA, AIR arithmetization over BabyBear, FRI-STARK proving, recursive composition, precompile acceleration, and decentralized proof markets — are likely to persist as the dominant approach through the near term, even as specific implementations evolve.

For a ZK engineering student, the most important takeaway is not the specific systems or benchmarks — those will change — but the underlying design principles: the tradeoffs between generality and efficiency, the role of arithmetization in converting computation to provable constraints, the distinction between succinctness and zero-knowledge, and the emerging ecosystem architecture of zkVMs, zkCoprocessors, and proof markets. These principles will remain relevant regardless of which specific zkVM dominates in five years.

For the ZK Developer

The best way to internalize zkVMs is to build with them. Start with RISC Zero's "Hello World" guest program or SP1's quickstart — prove a Fibonacci computation in under 30 minutes. Then explore precompiles by proving an ECDSA signature verification. Then read the RISC Zero ZIRGEN specification to understand what the prover is actually doing. The gap between reading about zkVMs and deeply understanding them closes quickly when you have a working proof in front of you.

SP1's ExecutionRecord holds one Vec per opcode type (add_events, sub_events, mul_events, bitwise_events, etc.). This is not incidental — it reflects the AIR architecture. Each chip only processes its own event type; chips run independently in parallel. The prover's throughput scales with the number of distinct operation types that can be proved concurrently, which is why GPU parallelism is so effective for SP1.

13.3 Program 1 — ZK Password Verifier

Concept

The first program demonstrates the fundamental zero-knowledge property: proving knowledge of a secret without revealing it. A SHA-256 hash of a private password is committed as a public output. The verifier receives only the hash; the password itself is never part of the proof. This models practical applications such as zkKYC (prove you passed identity verification without revealing your credentials) and anonymous authentication.

Guest Program

#![no_main] sp1_zkvm::entrypoint!(main); use sha2::{Digest, Sha256}; pub fn main() { // PRIVATE: read secret password — never committed to proof let password = sp1_zkvm::io::read::<String>(); // Hash it with SHA-256 inside the zkVM let mut hasher = Sha256::new(); hasher.update(password.as_bytes()); let hash = hasher.finalize(); // PUBLIC: commit only the hash — not the password sp1_zkvm::io::commit_slice(&hash); }

The SHA-256 computation uses SP1's patched version of the sha2 crate (patch-sha2-0.10.8-sp1-6.0.0), which routes the hash computation through SP1's SHA-256 precompile. Rather than executing SHA-256 as thousands of individual RISC-V instructions, the precompile handles the entire compression function in a single optimised circuit invocation — reducing cycle count by approximately 10× compared to a naïve implementation (see §4.5).

13.4 Program 2 — ZK Age Verifier

Concept

The second program models a real-world compliance scenario: proving that a person meets a minimum age requirement without revealing their actual birth year or age. This pattern directly applies to DeFi KYC, age-gated content platforms, and on-chain identity systems where revealing exact age constitutes a privacy violation.

Guest Program

#![no_main] sp1_zkvm::entrypoint!(main); pub fn main() { // PRIVATE — never in the proof let birth_year = sp1_zkvm::io::read::<u32>(); // PUBLIC — verifier agrees on these parameters let current_year = sp1_zkvm::io::read::<u32>(); // 2026 let minimum_age = sp1_zkvm::io::read::<u32>(); // 18 // Computation proven inside the zkVM let age = current_year.saturating_sub(birth_year); let is_eligible = age >= minimum_age; // PUBLIC outputs — actual age and birth year stay hidden sp1_zkvm::io::commit(&is_eligible); sp1_zkvm::io::commit(&minimum_age); sp1_zkvm::io::commit(&current_year); }

13.5 Program 3 — ZK Human Verifier

Concept

The third program extends the age verifier to a multi-criteria human eligibility system. Three independent checks are performed inside the zkVM — age threshold, credit score threshold, and citizenship whitelist membership — and the result of each is committed independently. The verifier learns exactly which checks passed and which failed, but not the underlying values. Five test subjects with deliberately varied profiles were used to demonstrate all failure modes.

Guest Program

Several practical observations emerged from this implementation work:

The host/guest separation is the key developer abstraction. In all three programs, the complex orchestration (setting up the prover, managing inputs, reading public outputs, verifying the proof) lives in the host and is never proved. Only the minimal computation that requires trustless verification — the SHA-256 hash, the comparison against thresholds, the whitelist lookup — lives in the guest. Keeping the guest lean directly reduces cycle count and proving time.

The precompile advantage is concrete. Program 1 uses SP1's SHA-256 precompile via the patched sha2 crate. The 2,748 cycle count would be approximately 28,000 cycles without the precompile — a 10× difference that directly maps to 10× slower proving. For production applications dominated by hashing or elliptic curve operations, choosing the right crate version (and ensuring the workspace patch applies) is a critical performance decision.

Proof generation time is dominated by prover initialization, not computation. All three programs required approximately 10 seconds to prove on CPU. The actual program is small (under 3,000 cycles). The initialization cost — loading the proving key, setting up the STARK parameters, running the FRI commitment — dominates at this scale. For larger programs (millions of cycles), the amortised cost per cycle becomes much lower, which is why zkVMs are best suited to batch processing of significant workloads.

The zero-knowledge property is architectural, not incidental. In all three programs, the private inputs are never committed, logged, or accessible to the verifier — not because we carefully avoided exposing them, but because the SP1 guest model enforces this structurally. sp1_zkvm::io::read() feeds data into the witness (private); sp1_zkvm::io::commit() exposes data in the journal (public). The boundary between private and public is explicit in the API. This architectural clarity is one of SP1's strongest developer experience contributions.

Implementation Summary

Three working zkVM programs were implemented and proved on SP1 v6.1.0, built from source. The programs demonstrate hash preimage secrecy, threshold proof without value revelation, and multi-criteria selective disclosure — covering the core zero-knowledge patterns described in §8.4. All source code and proof outputs are reproducible using the succinct Rust toolchain and CPU-mode proving with no GPU required.
