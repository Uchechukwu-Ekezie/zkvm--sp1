Theory · GMR 1989 + Practical SNARK/STARK Extensions

A zero-knowledge proof has four properties

  • 01
    Completeness.  If the statement is true, an honest prover can always convince an honest verifier. A correct claim is always provable.
  • 02
    Soundness.  If the statement is false, no cheating prover can produce a valid proof. Mathematical truth cannot be forged.
  • 03
    Zero-Knowledge.  The verifier learns nothing beyond the truth of the claim. No private data leaks — not even partially.
  • 04
    Succinctness.  The proof is small and verifiable in milliseconds, regardless of how long the original computation took. Added by SNARK/STARK constructions — the property that makes zkVMs practical at scale.

The mathematical core of a zkVM

Arithmetization — execution becomes polynomial constraints

After the executor traces every instruction, arithmetization converts that trace into polynomial equations. If any row is wrong, at least one polynomial fails to equal zero — the proof is rejected. This is how mathematics enforces computational correctness.

AIR — used by SP1

Algebraic
Intermediate Repr.

Constraints between adjacent rows:
P(rowᵢ, rowᵢ₊₁) = 0

Natural fit for step-by-step execution. SP1, RISC Zero, and Cairo all use AIR. The add.rs constraint we read in the source is exactly this.

R1CS — Groth16

Rank-1 Constraint
System

Each constraint:
(a·x) ⊙ (b·x) = (c·x)

Powers Groth16. Every branch and loop requires multiplicative overhead — inefficient for VM execution traces. Rarely used in modern zkVMs.

PLONKish — Halo2

Custom Gate
Arithmetization

Generalises R1CS with arbitrary custom gates per row. Efficient range checks, lookups, bitwise ops. Used by zkSync Boojum and Scroll.

Why AIR wins for RISC-V zkVMs: P(rowᵢ, rowᵢ₊₁) = 0 directly models "state at cycle i+1 follows correctly from state at cycle i." Every opcode gets one chip. SP1's AddChip enforces: result = op_b + op_c, PC += 4, clock += 8, register reads consistent. One wrong value anywhere → non-zero polynomial → proof rejected.

One of the hardest problems in zkVM design

Memory Checking — proving your program didn't cheat

Imagine a program that reads and writes to millions of memory slots. The question SP1 must answer mathematically: when the program read slot 42, did it get the correct value — the one that was last written there?

Think of it like a bank account.
Your balance must reflect the last transaction — not an old one.

SP1 stamps every memory operation with a clock number. Every read proves: "the value I got was written at clock T, and nothing was written to this address between T and now." One mathematical check at the end confirms the entire memory history is consistent.

How it works — step by step

① Every write is logged
Address + value + clock time → saved
② Every read records when it last saw a write
Address + value + clock of last write → saved
③ Rule: last write must be in the past
Clock of last write < Clock of this read ✓
④ One final check ties it all together
Every read matches exactly one write in the log — mathematically proven

The proof generation engine — Plonky3, SP1's backend

FRI-STARK Pipeline — how SP1 builds a proof

Step 1 — Trace Commitment

Interpolate the execution trace as polynomials over BabyBear. Commit via a Merkle tree over a low-degree extension field. The prover now holds a commitment to "this is what my program did."

Step 2 — Constraint Composition

Combine all AIR constraints into one quotient polynomial Q(x). Q(x) = 0 on all rows if and only if every instruction executed correctly. Q(x) is committed.

Step 3 — FRI Proximity Proof

The computationally heavy step. Prover convinces verifier Q(x) has low degree — it's a real polynomial, not arbitrary data. Achieved via FRI: repeated polynomial folding + Merkle openings.

Step 4 — Query Phase

Verifier sends random challenges. Prover opens committed polynomials at those points with Merkle proofs. 128-bit security requires ~50 queries. This finalises the binding.

Why BabyBear (p = 2³¹ − 2²⁷ + 1)?  Fits in a 32-bit word — no 64-bit overflow handling. CPUs and GPUs run 32-bit arithmetic at near-native speed. Goldilocks prime (64-bit): 2× overhead. BN254 (256-bit): ~10× overhead. BabyBear is a core engineering reason SP1 outperforms older systems.

The central design tension every zkVM must resolve

The zkVM Trilemma — no system achieves all three

Speed

Proving Performance

Fast proof generation. Real-time for Ethereum blocks. Requires efficient field arithmetic (BabyBear), GPU parallelism, precompiles, STARK-based parallelism.

Optimised by: SP1 Hypercube, Airbender, Brevis Pico.

Security

Proof Soundness

No trusted setup. Post-quantum secure. Formally verified circuits. Resistance to soundness bugs, range-check gaps, and memory consistency errors.

Optimised by: RISC Zero (ZIRGEN formal verification), Nexus (formal specification).

Generality

Developer Experience

Any program, any language, existing libraries. No custom DSL. Full RISC-V ISA including complex memory, branches, loops. Write normal Rust, get a proof.

Optimised by: SP1, RISC Zero. Cairo sacrifices this — you must write Cairo.

SP1's position: Maximum generality + strong speed via precompiles and GPU — at slightly higher per-cycle overhead than ZK-native ISAs. Jolt bets on lookup-based speed but sacrifices GPU support and production readiness. Cairo maximises speed but requires a custom language. The trilemma is why every architectural choice is a tradeoff — and why we chose SP1 for our implementation.

zkVM vs zkCoprocessor

"A zkVM is trustless AWS Lambda — run any code, get a proof. A zkCoprocessor is a blockchain API with proofs — pre-built, optimised for known tasks, plug and play."

What we built maps directly to the zkVM paradigm: Our three programs implement novel computation — password hashing, age threshold checking, multi-criteria KYC — with no pre-built coprocessor available. We had to write the guest logic and prove it directly on SP1. In a production DeFi system, a coprocessor like Axiom would handle high-frequency pre-defined queries faster and cheaper. Both paradigms are valid; the choice depends on how novel your computation is.

Precompile choice is a proving decision, not a software decision

SP1 Precompiles — 5–10× cycle reduction

SP1 intercepts specific function calls and replaces the naïve RISC-V trace with a hand-optimised AIR chip. The cryptographic semantics are identical — the proof is still correct. The cycle count plummets. Choosing the wrong crate version silently costs you the speedup.

OperationNaïve RISC-V cyclesWith SP1 precompileSpeedup
SHA-256 (64 bytes)~2,000~20010×

Our result: ZK Password Verifier without the precompile = ~28,000 cycles. With SP1's patched sha2 crate routing to the SHA-256 precompile = 2,748 cycles. A 10× reduction from one dependency version. Not an optimisation pass — a build configuration decision. This is what makes ZK programming different from normal software engineering.

Where zkVMs are deployed in production today — blockchain

Real-world applications — blockchain infrastructure

Layer 2 Scaling — zkRollups

zkSync Era, Polygon zkEVM, Scroll execute transactions off-chain and post one validity proof to Ethereum L1. One proof covers thousands of transactions, verified in milliseconds. Unlike optimistic rollups, there is no 7-day challenge window — cryptographic finality is instant. zkSync processes thousands of TPS vs Ethereum L1's 15.

Ethereum L1 Real-Time Proving

The Ethereum Foundation's target: prove each L1 block in under 12 seconds so validators can verify by checking a proof, not re-executing. First achieved October 2025 by Brevis Pico (6.9s, 64 GPUs). SP1 Hypercube: 10.3s. Airbender: 35s on a single H100. This would allow a 10–100× increase in L1 throughput.

Cross-Chain Bridges & Light Clients

Traditional bridges use multi-sig federations — trusted and hackable. zkVM light clients prove one chain's block headers to another with mathematics only, no committee. Celestia DA attestation proofs, IBC light client proofs, and EigenLayer AVS all use this pattern. Polygon and Avail use SP1 for cross-chain infrastructure.

Privacy Identity & Compliance — our pattern

zkKYC: prove you passed identity verification without showing your passport. Private voting: prove your vote is valid without revealing your choice. Anonymous DeFi lending: prove creditworthiness without revealing your balance sheet. This is exactly the pattern in our ZK Age and ZK Human Verifier — provable facts, zero data exposure.

Where zkVMs are deployed in production today — beyond blockchain

Real-world applications — outside of crypto

Verifiable AI Inference (ZKML)

As AI becomes critical infrastructure, a trust gap emerges: how does a user know a provider ran the claimed model on the claimed input without tampering? RISC Zero's zkVM has been used to prove transformer inference. Giza (built on Cairo) specialises in on-chain AI. Today: proving classification networks and recommendation engines. Trajectory: increasingly capable models as provers speed up.

On-Chain Gaming with Hidden State

zkVMs enable on-chain games where moves are proved valid without revealing the player's full board state. Chess, poker, strategy games. A player can prove "I moved my knight legally" without revealing where all their pieces are. Dark Forest (Ethereum) pioneered this category — a real-time strategy game where your location is a private ZK input. The entire game runs trustlessly on-chain.

Supply Chain & Data Provenance

A pharmaceutical company can prove a drug batch was stored at the correct temperature throughout its supply chain — without revealing commercially sensitive routing or supplier information. A financial institution can prove its trading algorithms executed within regulatory bounds — without revealing proprietary strategy. "Proof of correct computation over private data" is a general pattern applicable across industries.

Scientific Computing & Auditable Pipelines

Clinical trial results, financial model outputs, and government statistics all suffer from the same problem: results are published but the computation is trusted, not verified. zkVMs allow any computational pipeline to produce a proof of correct execution — auditors verify the proof, not the raw data. The data stays private; the result is mathematically guaranteed.

What the build actually taught us — things no slide covers

Lessons from building on real infrastructure

  • 01
    The precompile is a build dependency, not an optimisation.  We didn't "enable" it. We used SP1's patched sha2 crate, which routes automatically. The wrong crate version silently costs 10× more cycles. Nothing warns you.
  • 02
    SP1 updated from v6.0.0 to v6.1.0 mid-setup.  Missing Go for gnark FFI. Missing protoc for gRPC. These are real build errors from real infrastructure — not toy toolchain problems. We fixed every one. This is what "production software" actually means.
  • 03
    Reading add.rs closed the loop.  Every AIR constraint tutorial made sense abstractly. Line 202 of SP1's add.rs made it concrete — result = op_b + op_c, PC += 4, clock += 8. The gap between understanding and knowing only closes with primary source.
  • 04
    ZK correctness is a compiler property, not a runtime check.  If your program compiles and runs in SP1, the ZK boundary is already enforced. There is no "ZK mode" to enable. The API makes it impossible to get it wrong — which is the real engineering achievement.
